# VC 信息聚合 Agent — 系统设计文档

## 1. 问题背景

某头部投资机构的管理合伙人，每天需要跟踪 AI、芯片、机器人三个领域的行业动态。他的信息来源分散在 Twitter/X、YouTube、微信公众号、科技媒体等多个平台，面临三个核心痛点：

- **信息过载与噪音**：每天各平台产生数百条相关内容，其中大量是广告、水文、重复报道。人工筛选耗时 1-2 小时，严重挤压决策时间
- **跨平台、跨语言障碍**：英文推文、中文公众号、视频内容分散在不同平台，格式各异，无法统一浏览。中英文信息之间存在时差和信息差
- **缺乏投资视角的提炼**：原始信息是事实陈述，但 VC 需要的是"这件事对投资意味着什么"——需要从事实到洞察的转化

本系统的目标：构建一个 AI 驱动的信息聚合 Agent，自动完成"采集 → 过滤 → 摘要 → 推送"全流程，将每天 1-2 小时的信息筛选压缩到 5 分钟的简报阅读。

## 2. 架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                  Scheduler (Cron / APScheduler)              │
└──────────────────────────┬──────────────────────────────────┘
                           │
            ┌──────────────┼──────────────┐
            ▼              ▼              ▼
     ┌────────────┐ ┌────────────┐ ┌────────────┐
     │  YouTube   │ │    RSS     │ │  (扩展位)   │
     │ Collector  │ │ Collector  │ │ Twitter/X  │
     └─────┬──────┘ └─────┬──────┘ └─────┬──────┘
           └──────────────┼──────────────┘
                          ▼
               ┌─────────────────────┐
               │   Content Filter    │
               │ (规则评分 + 反馈权重) │
               └──────────┬──────────┘
                          ▼
               ┌─────────────────────┐
               │   LLM Summarizer    │
               │ (摘要 + 投资洞察)    │
               └──────────┬──────────┘
                          ▼
               ┌─────────────────────┐
               │  Brief Generator    │
               │ (Markdown + 飞书)   │
               └──────────┬──────────┘
                          ▼
               ┌─────────────────────┐
               │   Feedback Loop     │
               │ (用户反馈 → 权重更新) │
               └─────────────────────┘
```

| 模块 | 职责 | 实现方式 |
|------|------|---------|
| Scheduler | 定时触发采集 Pipeline | 系统 Cron / APScheduler |
| YouTube Collector | 频道订阅 + 关键词搜索双模式采集 | YouTube Data API v3 |
| RSS Collector | 从科技媒体 RSS 源拉取文章 | feedparser |
| Content Filter | 多维度质量评分 + 反馈权重调整 | 规则引擎 + JSON 偏好存储 |
| LLM Summarizer | 生成摘要 + "Why it matters" 投资洞察 | DeepSeek / OpenAI 兼容 API |
| Brief Generator | 输出 Markdown 简报 + 飞书推送 | 飞书 Webhook |
| Feedback Loop | 收集用户反馈，更新过滤权重 | 本地 JSON + CLI |

## 3. 数据采集方案

### 3.1 YouTube（已实现）

- **采集方式**：YouTube Data API v3，双模式并行
  - 频道订阅模式：预设 KOL 频道（a16z, AI Explained, Bloomberg Technology 等），通过 `playlistItems.list` 拉取最新视频
  - 关键词搜索模式：按领域关键词搜索，捕获长尾内容
- **技术限制**：每日 10,000 units 配额；无法直接获取完整字幕
- **应对方案**：频道模式 API 消耗低（playlistItems 仅 1 unit/次 vs search 100 units/次）；字幕通过 `youtube-transcript-api` 获取

### 3.2 RSS 聚合（已实现）

- **采集方式**：feedparser 解析 RSS/Atom 源
- **预设源**：TechCrunch, MIT Tech Review, Ars Technica（英文）；36氪, 量子位, 机器之心（中文）
- **优势**：零成本、无 API 限制、覆盖面广
- **限制**：RSS 内容通常只有摘要，全文需额外抓取；部分媒体 RSS 更新有延迟

### 3.3 Twitter/X（设计阶段）

- **采集方式**：Twitter API v2 Bearer Token
- **技术限制**：免费 tier 每月仅 1500 条读取，远不够覆盖多领域 KOL
- **MVP 不选它的原因**：免费额度太低，性价比差。建议后续用 Basic tier（$100/月，10,000 条）或通过 RSS 桥接工具间接获取
- **替代方案**：Nitter RSS 桥接（合规风险需评估）

### 3.4 Reddit（规划中）

- **采集方式**：Reddit API（免费，需 OAuth2）
- **高价值子版块**：r/MachineLearning, r/LocalLLaMA, r/robotics, r/semiconductor
- **优势**：社区讨论往往比新闻更早捕捉到技术趋势变化；upvote 机制天然做了一轮质量过滤
- **限制**：API rate limit 100 requests/min；内容偏技术讨论，需要 LLM 提炼投资相关信号

### 3.5 arXiv（规划中）

- **采集方式**：arXiv API（免费，无需认证）
- **价值**：论文是判断技术趋势的硬信号——当某个方向论文数量激增时，往往预示着 6-12 个月后的产业变化
- **采集策略**：按 cs.AI, cs.RO, cs.AR 等分类拉取，结合引用数和机构（Google, Meta, OpenAI 等）做初筛
- **限制**：论文内容专业度高，需要 LLM 做"翻译"——将学术语言转化为投资人能理解的商业含义

### 3.6 HuggingFace（规划中）

- **采集方式**：HuggingFace Hub API
- **价值**：Trending models/papers/spaces 反映开源 AI 社区的技术风向
- **限制**：偏技术向，对非 AI 领域（芯片、机器人）覆盖不足

### 3.7 微信公众号（设计阶段）

- **技术限制**：无官方公开 API，微信生态封闭，反爬严格
- **应对方案**：
  - 方案 A（推荐）：接入 RSS 聚合服务（WeRSS、feeddd），将目标公众号转为 RSS 源
  - 方案 B：搜狗微信搜索入口，稳定性差
  - 方案 C：企业微信 API + 自建消息转发，成本高但最稳定

### 平台选择总结

MVP 选择 YouTube + RSS 的原因：YouTube 有免费 API 且内容丰富（视频分析是差异化能力），RSS 零成本覆盖主流科技媒体。两者组合已能覆盖 80% 的高质量信息源。Twitter/X 因免费额度不足暂不实现，Reddit 和 arXiv 作为下一优先级扩展。

## 4. "高质量"定义与过滤策略

### 冷启动阶段（规则过滤）

采用多维度评分机制，总分 0-100，阈值 40 分以上进入简报：

| 维度 | 权重 | 规则 |
|------|------|------|
| 来源权威性 | 25% | 预设 KOL 白名单加分；营销号黑名单扣分 |
| 内容长度 | 15% | YouTube > 5 分钟、文章 > 500 字 |
| 互动指标 | 20% | 点赞/转发/评论数综合评估 |
| 关键词匹配 | 20% | 命中 AI/芯片/机器人领域关键词密度 |
| 时效性 | 10% | 24h 内发布加分 |
| 原创性 | 10% | 去重后的内容加分 |

**广告/水文过滤**：包含推广短链、UTM 参数、营销关键词（"限时优惠""免费领取"）的内容自动降权或过滤。

### 有反馈后的迭代策略

1. **短期**：用户标记"有用"的内容，提取来源、主题、关键词，动态调整权重
2. **中期**：积累 50+ 条反馈后，训练轻量分类器（logistic regression），输入为评分维度 + LLM embedding
3. **长期**：用 LLM 做 few-shot 质量判断，prompt 中注入用户历史偏好的正负样本

反馈信号：👍 = 正样本（权重 +1），👎 = 负样本（权重 -2，负反馈信号更强）。

## 5. 简报格式设计

### 设计原则

- **5 分钟可读完**：控制在 8-12 条精选内容
- **手机友好**：短段落、emoji 标记、无复杂表格
- **投资视角**：每条内容附 "Why it matters" 分析
- **可操作**：附原文链接，方便深入阅读

### 简报样例（Mock 数据）

```
# 📋 VC 每日简报 — 2026.04.24（周五）

> 今日共采集 156 条内容，精选 10 条高质量信息。

---

## 🤖 AI 领域（4 条）

### 1. OpenAI 发布 o3-pro，推理成本降低 60%
📺 YouTube · AI Explained · 14 分钟
OpenAI 发布 o3-pro 模型，代码生成准确率达 94.2%，API 定价为 GPT-4o 的 1.5 倍但推理效率提升 3 倍。
💡 Why it matters: 推理成本下降将加速 AI coding assistant 赛道洗牌，Cursor 等公司需快速建立差异化壁垒。
🔗 观看原视频

### 2. a]6z：为什么我们 All-in AI Agent
📝 RSS · TechCrunch
a16z 发布年度 AI 投资主题报告，核心观点：Agent 的壁垒不在模型能力，而在垂直场景的数据飞轮。
💡 Why it matters: 头部 VC 的投资主题转向直接影响创业公司的融资策略和估值逻辑。
🔗 阅读原文

---

📊 今日数据：采集 156 条 → 精选 10 条（入选率 6.4%）
💡 趋势洞察：AI Agent 商业化和推理成本下降是本周最强信号...
```

### 设计理由

1. **按领域分区**：用户可跳过不感兴趣的板块
2. **来源标识**（📺/📝 + 平台名）：一眼判断信息可信度
3. **"Why it matters"**：从事实到投资洞察的转化，这是 Agent 相比人工阅读的核心增值
4. **底部数据统计**：展示过滤力度，建立信任
5. **控制在 8 条**：每条 40 秒（含 Why it matters），8 条 ≈ 5 分钟

## 6. 当前限制与设计权衡

### 创始人/投资人动态追踪

题目要求追踪"Twitter/X 上的科技创始人和投资人动态"。当前通过 YouTube 频道订阅（a16z、YC、OpenAI 等官方频道）和 RSS（TechCrunch 报道）间接覆盖了头部人物的公开动态，但没有直接接入 Twitter API。

原因：Twitter API 免费 tier 每月仅 1500 条读取，无法覆盖多领域 KOL 列表。Basic tier（$100/月）可解决，但超出 MVP 预算。生产环境建议接入 Basic tier，按 KOL 列表定向拉取，配合 RSS 桥接工具（如 Nitter RSS）作为备选。

### 用户反馈闭环

系统已实现完整的偏好学习机制（feedback.py 存储反馈 → filter.py 动态调整权重），但飞书端的反馈入口受限于 webhook 能力——webhook 机器人无法接收用户点击事件。

当前方案：CLI 工具（`python feedback.py`）回放简报并标记反馈。

生产方案：创建飞书企业应用，使用交互卡片（Interactive Card）的按钮回调功能，用户点击 👍/👎 后飞书将事件 POST 到回调服务器，实现真正的一键反馈。这需要：飞书开放平台应用审批 + 部署回调服务（FastAPI）+ 公网地址。

### 关键词动态更新

当前关键词列表是静态配置（sources.yaml），无法自动跟踪热点变化（如某周突然火的 GPT image generation）。

规划方案：每次生成简报后，让 LLM 分析本次内容中出现的新兴关键词，自动建议追加到 sources.yaml。需要人工确认机制防止关键词漂移。

## 7. 未来改进方向

1. **arXiv 论文追踪**：自动监控 cs.AI/cs.RO 等分类的高引用论文，提前捕捉技术趋势拐点
2. **Reddit 社区情绪分析**：r/MachineLearning 等子版块的讨论热度变化可作为投资信号的先行指标
3. **向量数据库语义去重**：用 embedding 相似度替代简单的标题匹配，解决"同一事件不同报道"的去重问题
4. **Learning-to-rank 个性化排序**：基于用户反馈训练排序模型，实现千人千面的信息流
5. **实时突发事件告警**：对重大事件（融资、发布、政策变化）不等每日简报，即时推送飞书通知
